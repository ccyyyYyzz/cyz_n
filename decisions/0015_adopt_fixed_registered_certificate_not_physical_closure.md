# Decision 0015 — adopt the fixed registered certificate, not physical closure

## Verdict

The Brief 0014 software-evidence gap is closed for one exact registered
scheduled-kernel contract. The readable verifier executes, imports no
generator module, reconstructs every registered control through a source-
separated fixed implementation, binds the actual loaded source files, reruns
the external generator, checks real opaque-ID renaming, performs live `S9`
covariance and executes the hostile suite in fresh child processes.

This decision adopts a fixed-contract certificate. It explicitly rejects the
stronger interpretation that `constructor_spec.json` is a general executable
DSL or that the certificate derives a physical three-dimensional closure.

## Adopted facts

1. The canonical constructor specification, semantic projection and control
   manifest are fixed by three separately recorded digests.
2. The generator and source-separated verifier reconstruction produce equal
   canonical expanded SHA-256 digests and exact counts for all eleven
   controls. Subject to SHA-256 collision resistance, this binds the verifier
   reconstruction to the generator expansion.
3. Every registered row is normalized, every state has one row and schedule,
   positive accepted non-null atoms have declared reverse atoms, killed exits
   preserve their source physical projection, and the registered
   energy/charge/work ledgers close.
4. Deterministic finite-countdown schedule records are checked within the
   scheduled Markov-renewal representation; complete residual-age
   trajectories, CTMC semantics and detailed balance are not adopted.
5. On the verifier side, all eight adjacent `S9` generators are replayed live
   for the ordinary, three-large, four-large, arity-two and arity-four roles.
6. The arity-two and arity-four controls pass through the same fixed
   implementation, establishing that the code does not always substitute
   arity three.
7. The exact eleven-file executable inventory, normalized source hashes,
   repository root and actual loaded `__file__` paths are bound.
8. LF/CRLF and JSON formatting changes preserve the canonical artifact.
9. The 94 named hostile mutations are accepted as a live regression suite
   only when each child exits exactly 2 with a localized structured semantic
   error.
10. A PASS generation is published only after a
    pre-verifier-module-import in-progress sentinel, successful live execution
    and pre-/post-commit reverse checks that the artifact and executable
    sources did not change during the run. Side components are explicitly
    non-certificates. The stored main PASS is a generation-bound execution
    record under trusted repository provenance; `--check-report` checks only
    freshness and internal consistency, while independent live evidence
    requires rerunning `--report`.

## Claims not adopted

### General specification interpretation

The Python implementations do not execute an arbitrary
`constructor_program`, natural-language constitutive rule or parameter-
influence graph. The exact full-specification digest selects one audited
implementation on each side. A specification change is rejected rather than
interpreted.

The certificate may therefore be described as:

```text
one fixed registered contract
two source-separated reconstruction paths
one live finite evidence suite
```

It may not be described as a universal constructor interpreter or proof that
each possible serialized rule mutation has operational semantics.

The two model implementations are source-separated but deliberately
isomorphic. Their agreement is a differential software check, not an
independent derivation of the constitutive physics, and common-mode
algorithmic mistakes remain possible.

### Fundamental non-encoding

The fixed input boundary contains no response rank, target rank, visible
count, active mask or dimension label, and the source-level mutation suite
rejects representative hard-coded-triple and opaque-label branches. This is
fixed-release forbidden-input and source-binding evidence. It is not a theorem
over every possible implementation, source-family completion or hidden
variable.

### Physical three-selection

Frame arity remains an explicit microscopic input and the ordinary closure
uses arity three. The arity controls show software sensitivity; they do not
derive arity three from a physical source model.

The sparse impact law, deterministic return clock, history kernel, reverse
ratio, reservoir ledger and initial preparation remain proposed closure
choices. No strict-residence margin, visible-rank plateau, Lorentzian
signature or \(3+1\) selection theorem follows from this certificate.

Freeze the certificate layer except for discovered correctness defects. Move
the research effort to physical closure: derive or constrain the frame-role

loss-of-plateau prediction.
